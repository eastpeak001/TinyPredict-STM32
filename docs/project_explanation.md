# TinyPredict-STM32 项目答辩与自述文档

## 1. 一句话介绍

TinyPredict-STM32 是一个基于 STM32F103C8T6 和 MPU6050 的低成本设备振动监测节点，可以采集振动数据、计算 RMS、判断 NORMAL / WARNING / ALARM 状态，并通过 OLED、LED、串口和 Python 工具进行展示与分析。

## 2. 本项目解决的问题

本项目面向低成本设备振动/姿态状态监测和预警问题。对于小型风扇、电机、泵、旋转机构等设备，如果出现松动、偏心、异常振动，通常只能靠声音、手感或肉眼观察判断，不够直观，也不利于记录和复盘。

TinyPredict-STM32 的目标是做一个小型、低成本、可解释、可扩展的监测节点，实现：

- 采集设备三轴加速度数据。
- 通过高通滤波提取动态振动分量。
- 使用滑动 RMS 表征振动强度。
- 根据阈值、迟滞和连续计数判断状态。
- 在 OLED 和 LED 上本地显示状态。
- 通过 USART1 输出 CSV 数据。
- 使用 Python 记录数据、绘图并生成分析报告。

它不是工业级振动诊断仪，而是一个适合自动化专业学习、演示和继续扩展的原型系统。

## 3. 系统工作流程

```text
MPU6050 数据采集
        -> RMS 特征提取
        -> system_status 状态分类
        -> OLED / LED / UART 输出
        -> Python 数据记录
        -> 自动分析报告
```

具体流程如下：

1. `mpu6050.c` 通过 I2C1 读取 MPU6050 原始三轴加速度。
2. `sensor_task.c` 保存最新的 ax、ay、az 加速度数据，单位为 g。
3. `vibration_algo.c` 估计慢变化重力分量，提取动态振动分量，并计算滑动 RMS。
4. `system_status.c` 根据 RMS、迟滞阈值和连续计数防抖判断 NORMAL / WARNING / ALARM。
5. `oled_ui.c`、`alarm_task.c`、`debug_uart.c` 分别负责 OLED 显示、PC13 LED 报警和 USART1 CSV 输出。
6. `tools/serial_logger.py` 在电脑端保存串口 CSV 数据。
7. `tools/analyze_log.py` 生成 RMS 曲线、加速度曲线、状态统计图和 Markdown 分析报告。

## 4. 硬件角色说明

- STM32F103C8T6：主控芯片，负责传感器读取、算法计算、状态判断、OLED 显示、串口输出和 LED 控制。
- MPU6050：三轴加速度传感器，通过 I2C1 采集设备振动相关的加速度数据。
- OLED：0.96 寸 SSD1306 I2C 显示屏，用于现场显示项目名、RMS 和状态。
- USART1：串口调试和 CSV 数据输出通道，通过 USB 转 TTL 接到电脑，波特率 115200。
- PC13 LED：本地报警指示灯，低电平点亮，ALARM 状态时点亮。
- ST-LINK：用于 SWD 下载和调试固件。
- PCB：V0.4B 版本计划把 STM32F103C8T6 最小系统、电源、SWD、USART1、OLED、MPU6050、LED 等接口集成到自绘 PCB 上。

## 5. 固件模块说明

### mpu6050.c

`mpu6050.c` 是 MPU6050 的底层驱动。它使用 HAL 的 I2C 读写函数完成传感器唤醒、`WHO_AM_I` 检查、原始加速度读取，并把 raw 数据转换成 g 单位。

### sensor_task.c

`sensor_task.c` 是传感器任务层。它负责初始化 MPU6050，判断传感器是否可用，周期调用 `MPU6050_ReadAccel()`，并保存最新的 `ax`、`ay`、`az`。

### vibration_algo.c

`vibration_algo.c` 是振动算法模块。它通过低通滤波估计慢变化重力分量，再用原始加速度减去重力估计值，得到动态振动分量，最后计算滑动 RMS。

### system_status.c

`system_status.c` 是系统状态管理模块。它统一管理 NORMAL、WARNING、ALARM 的判断逻辑，集中保存阈值、迟滞和连续计数防抖规则，避免 OLED、串口和 LED 各自维护一套状态判断。

### alarm_task.c

`alarm_task.c` 负责 PC13 LED 报警控制。校准阶段 LED 保持熄灭；当 `system_status` 判断为 ALARM 时，PC13 输出低电平点亮 LED。

### oled_ui.c

`oled_ui.c` 负责 SSD1306 OLED 显示。显示内容包括 TinyPredict、RMS 数值、当前状态和版本信息。OLED 刷新周期受到限制，避免 I2C 占用过高。

### debug_uart.c

`debug_uart.c` 负责 USART1 文本和 CSV 输出。它输出 CSV 表头和数据行，当前字段为 `time_ms,ax,ay,az,rms,status`。

### app_main.c

`app_main.c` 是应用层调度模块。它把 `main.c` 保持为 CubeMX/HAL 初始化层，自己负责按周期调用传感器更新、算法更新、状态更新、报警更新、OLED 刷新和串口输出。

## 6. 算法说明

MPU6050 采集的是三轴加速度：`ax`、`ay`、`az`。但是加速度里面既包含设备振动，也包含重力影响。如果直接用原始 ax、ay、az 判断振动，模块慢慢倾斜时也会让数据变化，容易误判为异常振动。

为了解决这个问题，算法先用低通滤波估计慢变化的重力分量。慢变化部分可以理解为姿态和重力的影响。然后用原始加速度减去重力估计值，得到动态振动分量：

```text
振动分量 = 原始加速度 - 重力估计分量
```

之后，算法计算振动幅值，并对最近一段数据做滑动 RMS。RMS 可以理解为一段时间内振动能量的强弱，数值越大，说明振动越明显。

系统状态分为三类：

- NORMAL：振动较低，设备状态稳定。
- WARNING：振动高于正常范围，可能存在轻微异常。
- ALARM：振动明显偏高，需要重点关注。

为了避免状态在阈值附近频繁跳变，系统加入了迟滞和连续计数防抖。

当前状态规则：

- 进入 WARNING：`rms >= 0.05`，连续 3 次。
- 退出 WARNING：`rms < 0.04`，连续 5 次。
- 进入 ALARM：`rms >= 0.15`，连续 3 次。
- 退出 ALARM：`rms < 0.12`，连续 5 次。

迟滞的意义是进入阈值和恢复阈值不同，避免 RMS 在边界附近上下波动时状态来回跳。连续计数防抖的意义是要求异常持续一小段时间才改变状态，减少偶发噪声造成的误报警。

## 7. 数据记录与分析

固件通过 USART1 输出 CSV 数据，当前字段为：

```text
time_ms,ax,ay,az,rms,status
```

字段说明：

- `time_ms`：系统运行时间，单位 ms。
- `ax`、`ay`、`az`：三轴加速度，单位 g。
- `rms`：滑动 RMS 振动特征。
- `status`：当前状态，包含 NORMAL、WARNING、ALARM，也可能在启动阶段显示 CALIBRATING。

`tools/serial_logger.py` 用于读取串口数据，并保存为 CSV 文件。它可以过滤 OLED 初始化提示、MPU6050 初始化失败提示等非数据行。

`tools/analyze_log.py` 用于读取 CSV 文件并自动生成：

- RMS 曲线图。
- ax / ay / az 三轴加速度曲线图。
- 状态统计图。
- Markdown 实验报告。

保存数据和画图的原因是：一个监测系统不能只看瞬间显示，还需要能回放、比较和解释实验过程。曲线可以帮助判断 RMS 变化是否和真实振动情况一致，也方便答辩和后续优化。

## 8. 我的设计选择

- 为什么选择 STM32：STM32F103C8T6 成本低、资料多、适合教学和调试，同时具备 I2C、USART、SWD 等本项目需要的外设。
- 为什么选择 MPU6050：MPU6050 价格低、容易购买、I2C 接口简单，适合第一版振动监测原型。
- 为什么使用 RMS：RMS 简单直观，可以反映一段时间内振动强度，计算量也适合 STM32F103C8T6。
- 为什么做三状态：NORMAL / WARNING / ALARM 符合工业监测中的分级预警思路，也方便 OLED、LED 和串口显示。
- 为什么加入迟滞和防抖：真实传感器数据存在噪声，单次超过阈值就切换状态会导致显示和报警不稳定。
- 为什么增加 Python 分析工具：串口实时显示只能看当前状态，Python 工具可以保存数据、绘图和生成报告，方便实验复盘。
- 为什么做 PCB bring-up checklist：自绘 PCB 首次上电风险较高，分步骤检查可以降低短路、反接、焊接错误带来的损坏风险。

## 9. 答辩问题与参考回答

### Q1：这个项目主要实现了什么功能？

A：它实现了基于 STM32 和 MPU6050 的设备振动监测，能够计算 RMS，判断 NORMAL / WARNING / ALARM，并通过 OLED、LED、串口和 Python 工具展示与分析。

### Q2：为什么选择 STM32F103C8T6？

A：因为它成本低、应用广、资料多，支持 I2C、USART、SWD 调试，性能足够完成本项目的采集、计算和显示任务。

### Q3：为什么选择 MPU6050？

A：MPU6050 是常见的三轴加速度传感器，价格低，I2C 接口简单，适合做低成本振动监测原型。

### Q4：为什么不能直接用 ax、ay、az 判断振动？

A：因为 ax、ay、az 中包含重力分量。模块慢慢倾斜时，重力在三个轴上的投影会变化，容易造成误判。

### Q5：如何区分姿态变化和振动？

A：算法用低通滤波估计慢变化的重力分量，再用原始加速度减去重力分量，得到动态振动分量。

### Q6：RMS 在项目中代表什么？

A：RMS 代表一段时间内振动强度的均方根值，可以理解为振动能量强弱的特征。

### Q7：为什么使用滑动 RMS？

A：滑动 RMS 可以让状态判断基于最近一段数据，而不是单个瞬时点，结果更加稳定。

### Q8：系统有哪些状态？

A：启动阶段有 CALIBRATING，运行阶段有 NORMAL、WARNING 和 ALARM 三种状态。

### Q9：当前状态阈值是多少？

A：进入 WARNING 的核心阈值是 RMS 大于等于 0.05，进入 ALARM 的核心阈值是 RMS 大于等于 0.15。

### Q10：什么是迟滞？

A：迟滞就是进入状态和退出状态使用不同阈值。例如进入 WARNING 是 0.05，但恢复 NORMAL 要低于 0.04。

### Q11：为什么要加连续计数防抖？

A：为了避免单个噪声点造成状态跳变，只有连续多次满足条件才切换状态。

### Q12：PC13 LED 的作用是什么？

A：PC13 LED 用于 ALARM 本地报警指示。它是低电平点亮，ALARM 时输出 GPIO_PIN_RESET。

### Q13：OLED 显示什么内容？

A：OLED 显示 TinyPredict、RMS 数值、当前状态和版本信息。

### Q14：USART1 输出什么格式？

A：USART1 输出 CSV 格式：`time_ms,ax,ay,az,rms,status`。

### Q15：为什么使用 CSV？

A：CSV 简单通用，方便串口记录，也方便用 Python、Excel 或其他工具分析。

### Q16：serial_logger.py 的作用是什么？

A：它读取串口输出，解析有效 CSV 数据，并保存到 `docs/data/` 目录中。

### Q17：analyze_log.py 的作用是什么？

A：它读取 CSV 数据，生成 RMS 曲线、加速度曲线、状态统计图和 Markdown 分析报告。

### Q18：app_main.c 的作用是什么？

A：它是应用层调度模块，负责按周期调用传感器、算法、状态、报警、OLED 和串口输出模块。

### Q19：这个项目有没有做频谱分析？

A：当前版本没有做 FFT 或频谱分析，目前主要用 RMS 表征振动强度。频谱分析可以作为后续升级方向。

### Q20：这个项目的局限性是什么？

A：MPU6050 不是工业级传感器，阈值来自有限测试，安装方式会影响结果，目前还不能识别具体故障类型。

### Q21：PCB bring-up checklist 的意义是什么？

A：它用于 PCB 焊接后的首次上电和分阶段验证，降低短路、反接、焊接错误导致损坏的风险。

### Q22：如何验证系统有效？

A：可以观察 OLED 状态、PC13 LED、串口 CSV 数据、RMS 随振动变化情况，并用 Python 工具生成曲线和报告。

## 10. 我的收获

通过这个项目，我学习了从嵌入式采集到数据分析的完整流程。

在嵌入式开发方面，我学习了如何使用 STM32 HAL，如何把 CubeMX 生成的初始化代码和自己的应用逻辑分开，如何设计 `app_main.c` 和多个功能模块。

在传感器采集方面，我学习了加速度数据不仅包含振动，还包含重力、姿态和安装方式的影响。

在 I2C 通信方面，我学习了 STM32 如何通过 I2C1 同时连接 MPU6050 和 OLED，并理解了 SCL、SDA、地址和上拉电阻的重要性。

在串口通信方面，我学习了如何通过 USART1 输出结构化 CSV 数据，并使用 USB 转 TTL 在电脑端记录。

在数据分析方面，我学习了如何用 Python 保存数据、绘制 RMS 曲线、绘制加速度曲线、统计状态并生成报告。

在状态判断方面，我学习了单纯阈值判断的不足，也理解了迟滞和连续计数防抖在实际传感器系统中的作用。

在 PCB bring-up 方面，我认识到自绘 PCB 不能直接一次性全功能上电，而应该先检查短路、电源、SWD、串口、I2C 和各个模块。

在工程文档方面，我学习到一个项目不仅要有代码，还需要 README、接线说明、算法说明、测试报告、PCB 检查清单和答辩说明，这样项目才更容易复现、展示和继续迭代。
